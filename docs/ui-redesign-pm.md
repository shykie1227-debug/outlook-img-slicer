# outlook-img-slicer UI 重构产品分析

> 分析时间：2026-05-09
> 分析依据：SPEC.md (v3.0.20260509) + main.py (V3.0.20260511-TEST)

---

## 1. 当前结构问题诊断

### 1.1 严重不符项（SPEC vs 实现）

| 问题 | SPEC 定义 | 实际实现 | 风险 |
|---|---|---|---|
| 切片阈值 | 1728px | 1200px (`MAX_HEIGHT_PER_SLICE = 1200`) | 用户拿到不符合承诺的结果 |
| 快捷键 | Ctrl+V / Ctrl+O / Esc | **完全未实现** | 承诺的功能缺失 |
| Outlook 未安装提示 | 明确提示 | 无专门检测，直接报错 | 用户体验差 |

### 1.2 操作流程问题

**当前流程：**
```
拖入/选择文件 → [显示宽度设置] → [收件人/标题] → 切片处理 → [创建邮件]
                                                                ↓
                                                          保存切图（可选）
```

**问题 1：「显示宽度」暴露时机错误**
- 切片方案生成后，用户才能知道要切几张图
- 但「显示宽度」在文件加载前就可以修改
- 用户在不理解业务逻辑的情况下改这个值，只会增加困惑
- 而且：CSS `width` 参数的正确值用户根本无从判断

**问题 2：收件人/标题是"谎言"UI**
- SPEC 中完全没有提到这两个字段
- 实现中确实收集了值，但 `outlook_sender.py` 用了吗？main.py 里 `create_email_with_images(html_content, subject, to_addr, ...)` 传了这些参数
- 如果 `to_addr` 传给了 Outlook 但 Outlook 并没有 prefill 收件人，用户会认为工具坏了
- 如果 `subject` 没有写入，标题字段是多余的装饰

**问题 3：流程中有价值信息太少**
```
用户看到：收件人邮箱（可选）  邮件标题（可选）  显示宽度 960px
用户不知道：切片数量、原始尺寸、为什么这样切
```

### 1.3 信息层级问题

当前布局（从上到下）：
1. Header + 副标题（静态装饰，无操作价值）
2. 拖放区域
3. 选择文件 / 重置 / **显示宽度** ← 辅助信息优先级过高
4. 收件人邮箱 ← 辅助信息
5. 邮件标题 ← 辅助信息
6. 预览缩略图
7. 进度条
8. 状态文字
9. 创建 Outlook 邮件 / 保存切图

**问题：用户决策最需要的信息（切片数量 × 切片尺寸）不在显眼位置。**

### 1.4 用户心智模型

| 概念 | 用户能理解吗？ | 当前 UI |
|---|---|---|
| 1728px 限制 | ❌ 不知道 | ❌ 完全没有体现 |
| 为什么要切片 | ❌ 不知道 | ❌ 没有任何说明 |
| 显示宽度 960px 是什么意思 | ❌ 不知道 | ✅ 作为输入框显示，但没有帮助 |
| 收件人/标题 | ⚠️ 以为工具会帮他填 | ❌ 实际不填或不确定 |

## 2. 核心操作路径验证

### 2.1 主路径分析

```
拖入文件 → 创建邮件  (最短路径)
```

**验证结果：** 成立。拖入文件后，点击「创建 Outlook 邮件」即可。

**但存在问题：**
- 没有快捷键（Ctrl+O / Ctrl+V / Esc），用户不知道可以用键盘
- 切片数量超过 20 时需要确认，但这个提示只在 SPEC 里有，实现中有没有需要验证

### 2.2 次路径：保存切图

**验证结果：** 需要单独的「保存切图」按钮是合理的。

**但问题：**
- 这个按钮和「创建 Outlook 邮件」平级，视觉权重相当
- 实际上「创建 Outlook 邮件」才是主操作，「保存切图」是辅助的"万一不需要发邮件"功能
- 当前按钮样式：创建邮件是蓝色粗体 + 大按钮；保存切图是灰色普通按钮 → **已经是正确的层级区分**

### 2.3 辅助功能简化可能性

| 字段/功能 | 是否可简化 | 理由 |
|---|---|---|
| 显示宽度 | ✅ 可简化为默认值隐藏 | 大多数用户不需要改；改了也不知道对不对 |
| 收件人邮箱 | ✅ 可删除或改为"在 Outlook 中填写" | 工具无法保证写入 Outlook 收件人 |
| 邮件标题 | ✅ 可删除或改为"在 Outlook 中填写" | 同上 |
| 重置按钮 | ✅ 可简化为"Esc 关闭" | 重置操作低频；Esc 已能清状态 |
| 调试信息行 | ❌ 上线必须删除 | BUILD 版本信息暴露给用户 |
| 副标题 | ⚠️ 可缩短或删除 | 信息重复：拖放区已有提示 |

## 3. 按钮合理性分析

### 3.1 当前按钮列表

| 按钮 | 类型 | 位置 | 当前状态 |
|---|---|---|---|
| 选择文件 | 次操作 | 主区域 | 正常 |
| 重置 | 辅助 | 主区域 | 可疑（低频操作） |
| 创建 Outlook 邮件 | **主操作** | 底部 | ✅ 正确 |
| 保存切图 | 次操作 | 底部 | ✅ 正确 |

### 3.2 建议的按钮层级

```
[主操作]  蓝色大按钮，视觉最突出
          → 创建 Outlook 邮件（当前 ✅ 正确）

[次操作]  灰色中等按钮，与主操作拉开距离
          → 保存切图（当前 ✅ 正确）
          → 选择文件（当前 ✅ 正确，但可考虑降级）

[辅助]    文字链接或极小按钮
          → 重置（建议改为：操作区右上角小图标 或 去掉）
```

### 3.3 「重置」按钮是否必要

**结论：可以不要，理由：**
1. Esc 快捷键已覆盖「取消→重置」场景
2. 用户放错文件后，直接放新文件就覆盖了
3. 重置按钮在界面占了一个位置，但使用频率极低
4. 如果保留，应该给一个更隐晦的位置（如右上角小图标）

## 4. 精简版产品结构建议

### 4.1 简化后的 UI 元素列表

**保留（必要）：**
- [x] 拖放区域（承载核心交互入口）
- [x] 切片预览缩略图（让用户确认切片数量）
- [x] 状态文字（告知用户当前在做什么）
- [x] 进度条（给长时间操作反馈）
- [x] 创建 Outlook 邮件按钮（主操作）
- [x] 保存切图按钮（次操作）

**删除或降级（非必要）：**
- [ ] 收件人邮箱字段（功能不成立，或在 Outlook 窗口中填写）
- [ ] 邮件标题字段（同上）
- [ ] 显示宽度（改为内部逻辑，必要时才暴露）
- [ ] 重置按钮（Esc 覆盖）
- [ ] 调试信息行（上线必须删）
- [ ] 副标题（与拖放区提示重复）

### 4.3 简化后的操作流程

```
启动 → 拖入/选择文件 → 显示切片方案 → [创建 Outlook 邮件] 或 [保存切图]
                                   ↑
                            不需要任何中间步骤
```

**最短路径：2 步**（选文件 → 点发送）

### 4.4 关键改动优先级

| 优先级 | 改动 | 理由 |
|---|---|---|
| **P0** | 修复切片阈值为 1728px（与 SPEC 一致） | 数据正确性 |
| **P0** | 实现快捷键 Ctrl+O / Ctrl+V / Esc | SPEC 承诺 |
| **P0** | 删除调试信息行 | 正式发布不能有 |
| **P1** | 删除收件人/标题字段或明确其作用 | 避免用户误解 |
| **P1** | 切片信息改为显眼展示（切片数量 × 尺寸） | 用户需要确认 |
| **P2** | 将显示宽度从 UI 移除，改为内部固定值 | 用户无需修改 |
| **P2** | 移除重置按钮，以 Esc 替代 | 低频占用空间 |
