## Bug 1：切图残留 hotspot_map

### 现象
用户切图 A（标注 hotspot） → 拖入新图 B → hotspot_map 残留 A 的 hotspot（key="a1.png" 等）

### 根因
`_on_processed`（处理新切图完成的回调）只更新了 `slice_paths` 和 `slice_source_index`，**没清空 hotspot_map**。

```python
# main.py _on_processed
def _on_processed(self, paths: List[str]):
    self.slice_paths = paths
    self.slice_source_index = {os.path.basename(p): float(i + 1) for i, p in enumerate(paths)}
    # ❌ 缺这行：self.hotspot_map.clear()
    self.progress_bar.hide()
    ...
```

### 修复
```python
# V4.6.9 修复
self.hotspot_map.clear()  # 重新切图时必须清空
```

### 影响
- 表面看 hotspot_map 错乱，但实际不影响 B 发送（B 的文件名在 hotspot_map 里查不到 → 不会被错误切割）
- 主要影响用户认知："为什么我切了新图，旧图的 hotspot 还在？"

## Bug 3：多段拼接宽度按原图比例缩放

### 现象
V1 物理切割后，1 张原图被切成 N 段（如 1 个 hotspot → 3 段），邮件里视觉上总宽 = N × display_w，**远超原图按 display_w 缩放后的实际宽度**。

### 根因
`_build_image_row` 不看原图宽，每段都按 `display_w` 渲染：

```python
# 修复前
def _build_image_row(slice_path, cid, display_w, ...):
    actual_w, actual_h = _get_img_dimensions(slice_path)
    display_h = round(actual_h * display_w / actual_w)  # ❌ display_w 直接用
    ...
```

举例：1000×500 原图 + 1 hotspot (400~600) → 切 3 段 (400+200+400)，每段都按 650 渲染 → 邮件总宽 1950px（错的）。

### 修复
给 `SliceItem` 加 `original_width` 字段，**按原图宽比例分配显示宽度**：

```python
# V4.6.9 修复
if original_width > 0 and actual_w > 0:
    ratio = actual_w / original_width
    seg_display_w = round(display_w * ratio)  # ✓ 按比例
else:
    seg_display_w = display_w  # 兜底
```

### 效果
| 项 | 之前 | 现在 |
|----|------|------|
| 1000×500 + 1 hotspot 切 3 段 | 段宽 650+650+650 = 1950 | 段宽 260+130+260 = **650** ✓ |

### main.py 集成
```python
# main.py _build_slices_with_hotspots
# 用 sort_key 整数部分反推 source_index → 找到原切片 → 读 PIL.Image.size
source_idx = int(sk)
orig_path = si_to_orig.get(float(source_idx), '')
orig_w = orig_w_cache.get(orig_path, 0)
items.append(SliceItem(..., original_width=orig_w))
```

## Bug 4：HTML 横向拼接消除"碎片化"

### 现象（最严重）
**1 张原图标 1 个 hotspot → 邮件里视觉上是 3 张独立图片垂直堆叠，不是 1 张原图**。

### Step 1-5 调查过程

#### Step 1: 打印最终产物
| path | sort_key | width | height | href |
|------|----------|-------|--------|------|
| hs_001.png | 1.001 | 400 | 500 | None |
| hs_002.png | 1.002 | 200 | 500 | https://test.com |
| hs_003.png | 1.003 | 400 | 500 | None |

✅ 切割正确，sort_key 严格递增

#### Step 2: 拼接顺序
按 sort_key 排序：hs_001 → hs_002 → hs_003

#### Step 3: 总宽 = 原图
总宽 400+200+400 = 1000 ✓
高一致 500 = 500 ✓

#### Step 4: HTML 实际结构（**根因命中**）
```html
<tr><td><img width="260"></td></tr>   ← 第 1 张图（行 1）
<tr><td><a><img width="130"></a></td></tr>  ← 第 2 张图（行 2）
<tr><td><img width="260"></td></tr>   ← 第 3 张图（行 3）
```

**3 个 `<tr>` 纵向堆叠 = 3 张图垂直排列 = 1500px 高**（不是 500px）

正确应该是：
```html
<tr>
  <td><img width="260"></td>           ← 左段
  <td><a><img width="130"></a></td>   ← 中段（链接）
  <td><img width="260"></td>           ← 右段
</tr>
```

#### Step 5: 根因
V1 物理切割产物（每段 y1=0, y2=H 整高竖条）在 HTML 输出层**每段独占 1 个 `<tr>`** → 纵向堆叠。

### 修复

**拆 `_build_image_row` 为 `_build_cell`（返回 `<td>`），新增 `_group_by_source()` 按 sort_key 整数部分（=source_index）分组，同组 1 个 `<tr>` 多 `<td>` 横向拼接**。

```python
# html_assembler.py
def _group_by_source(slices):
    sorted_slices = sorted(slices, key=lambda s: s.sort_key)
    groups = []
    current_group = []
    current_source = None
    for s in sorted_slices:
        src_idx = int(s.sort_key)  # 1.001 → 1
        if current_source is None or src_idx != current_source:
            if current_group:
                groups.append(current_group)
            current_group = [s]
            current_source = src_idx
        else:
            current_group.append(s)
    if current_group:
        groups.append(current_group)
    return groups

def assemble_html(slices, display_w=650):
    sorted_slices = sorted(slices, key=lambda s: s.sort_key)
    groups = _group_by_source(sorted_slices)
    rows = ""
    for group in groups:
        cells = ""
        for s in group:
            cells += _build_cell(s.path, f"cid:slice_...", display_w, s.href, ...)
        rows += f'<tr>\n{cells}</tr>\n'  # ✓ 同组横向
    return html_template.format(rows=rows)
```

### 效果对比
| 场景 | 之前 | 现在 |
|------|------|------|
| 1 张原图 + 1 hotspot | 3 个 `<tr>` 纵向 | **1 个 `<tr>` 3 个 `<td>` 横向** |
| 2 张原图（智能切图 Y 切） | 6 个 `<tr>` 纵向 | **2 个 `<tr>`（每原图 1 行）** |
| Outlook 视觉 | 3 张图垂直排列 | 1 行图 = 完整原图 |

## 架构变更：V1 物理切割

### 背景
V4.6.5 之前用 `<map>+<area>` 给图片加可点击区域，但 Outlook Desktop 用 Word 引擎渲染，**不解析 `<map>/<area>`** → 桌面端客户点不动。

V4.6.6 改为 V1 物理切割：把原图按 hotspot X 范围切成纵向条状，**链接竖条用 `<a><img></a>` 包裹**（HTML 最基础结构，所有 Outlook 客户端可点）。

### V1 架构硬约束
- ✗ 仅纵向切割（X 轴），不切横向（Y 轴）
- ✗ 不支持重叠 hotspot（X 范围相交 → 拒绝）
- ✗ 不支持嵌套 hotspot（X 范围包含 → 拒绝）
- ✗ 不支持一个竖条挂多个 URL
- ✗ 不支持 `<map>/<area>` 兜底
- ✗ 不支持透明覆盖层

### 优势
- ✓ Outlook Desktop 100% 可点击
- ✓ 图上看不到任何标注（hotspot 边界不渲染）
- ✓ 多 hotspot 支持（任意位置）
- ✓ 切割线模型可扩展到 V2

### 核心代码
```python
# hotspot_slicer.py
def compute_cut_lines(img_w, hotspots):
    lines = {0, img_w}
    for h in hotspots:
        lines.add(h.x1)
        lines.add(h.x2)
    return sorted(lines)

def slice_image_with_hotspots(img, hotspots, source_index=0.0):
    cut_lines = compute_cut_lines(img.size[0], hotspots)
    stripes = []
    for i in range(len(cut_lines) - 1):
        x1, x2 = cut_lines[i], cut_lines[i + 1]
        stripe_img = img.crop((x1, 0, x2, img_h))  # 整高
        stripes.append(CutStripe(
            image=stripe_img, x1=x1, x2=x2,
            y1=0, y2=img_h,  # 整高
            sort_key=source_index + (i + 1) * 0.001
        ))
    return stripes
```

## 架构变更：V4.6.7 排序架构

### 背景
V4.6.6 物理切割后，HTML 输出顺序依赖隐式状态：
1. 智能切图产物 `image_slicer.detect_and_slice` 的返回顺序
2. `hotspot_slicer.slice_paths_by_hotspots` 的 append 顺序
3. `html_assembler` 的 for 循环顺序
4. 未来可能引入的 OCR / 文本切片 → 顺序来源越多越乱

V4.6.7 引入**显式 sort_key 字段**作为 HTML 输出的唯一排序依据。

### 数值层级
```
原切片:    sort_key = N.000   (N = 1, 2, 3, ...)
Hotspot 派生竖条: sort_key = N.M   (M = N*0.001, N=1, 2, ..., 999)
```

### 唯一排序点
```python
# html_assembler.assemble_html
sorted_slices = sorted(slices, key=lambda s: s.sort_key)
# 之后所有逻辑只用 sorted_slices
```

### 架构红线（禁止的依赖）
- ✗ 不允许依赖 append 顺序
- ✗ 不允许依赖文件名排序
- ✗ 不允许依赖目录遍历顺序
- ✗ 不允许依赖 `image_paths.index()` 兜底（O(N) 性能 + 重复路径 bug）

### 修复的 5 个 V4.6.7 隐藏 bug（V4.6.8 专家审查）
1. `Hotspot.normalized()` 不传 source_index（隐藏根因）
2. `HotspotMap.update()` 不透传 source_index
3. `HotspotMap.from_json()` 不透传 source_index
4. `slice_paths_by_hotspots` 兜底用 O(N) `.index()`
5. `reset_app()` 不重置 `slice_source_index`

详见 [架构-V4.6.7排序架构.md](./架构-V4.6.7排序架构.md)

## 调查方法论：gstack Iron Law

### 引用
> **NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST.**
> Fixing symptoms creates whack-a-mole debugging.
> —— `gstack-openclaw-investigate` SKILL.md

### 4 阶段流程
1. **Investigate**：收集症状、读代码、检查最近变更
2. **Analyze**：形成假设（基于实际证据，不是猜）
3. **Hypothesize**：列出可能根因 + 证据 + 验证方法
4. **Implement**：修复后跑测试 + 复现原症状

### 实际应用（V4.6.9 修复"碎片化"）
| 阶段 | 行为 |
|------|------|
| Investigate | 读 main.py / html_assembler.py / hotspot_slicer.py |
| Analyze | 用户报告"碎片错乱" → 假设有 4 级嫌疑（hotspot_slicer / sort_key / 切片高度 / HTML 结构） |
| Hypothesize | Step 4 打印 `<tr>` 数量 = 3，`<td>` 数量 = [1,1,1] → 命中第 4 嫌疑 |
| Implement | 拆 `_build_image_row` 为 `_build_cell` + 按 source_index 分组 |

### 5 步调查协议
- Step 1：打印最终产物（path/sort_key/width/height/href）
- Step 2：生成切片示意图 + 拼接顺序
- Step 3：验证所有切片宽高总和 = 原图
- Step 4：验证 HTML 实际结构（横向 vs 纵向）
- Step 5：定位最终根因（不修代码，先报告）
