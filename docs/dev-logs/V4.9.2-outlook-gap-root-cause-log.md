# V4.9.2 Outlook 拼接缝根因排查日志 (2026-06-18)

## 触发

用户反馈：发送到 Outlook 的长图中间仍有缝隙；该问题已修改几十次仍未解决；V3.0 版本反而没有该情况。要求彻底排查。

## 调试策略

进入 `superpowers-systematic-debugging` 的 3+ 失败架构审查路径：停止继续叠加 mso/height/像素补丁，先回溯 V3.0 正常实现并对比当前 V4.9.1。

## 关键证据

### V3.0 正常结构

V3.0 的 `html_assembler.py` 普通发送路径是：

```html
<td>
  <img src="cid:slice_001" width="650" style="display:block; ..." />
  <img src="cid:slice_002" width="650" style="display:block; ..." />
  <img src="cid:slice_003" width="650" style="display:block; ..." />
</td>
```

特点：
- 一个外层 table / tr / td。
- 切片之间没有 div/table/tr/td 边界。
- img 只有 width，无显式 height。
- 用户历史实测该结构无缝。

### V4.9.1 异常结构

对 3 张普通切片生成 HTML 的结构计数：

```text
div count   = 3
table count = 4
tr count    = 4
td count    = 4
img count   = 3
```

也就是每张切片外层都有：

```html
<div>
  <table>
    <tr>
      <td>
        <img>
```

即使 V4.9.1 已屏蔽 hotspot，该普通链路仍走复杂 wrapper。Outlook Word 引擎最容易在 block/table 边界插入不可控间隙。

## 根因判断

此前 V4.8.x 一直把根因假设锁在：

- PNG 物理高度
- HTML `<tr height>` / `<img height>`
- px→pt 四舍五入
- 4px 对齐
- materialize 预渲染

但 V3.0 无缝结构恰恰没有 per-slice `<tr height>` / `<img height>`。因此旧测试和旧修复方向锁死了失败架构：不断修每片 table row 的高度，而不是移除 Outlook 会造缝的 table/block 边界。

## 修复

V4.9.2 只做一个变量变更：

普通无热点链路（每个 group 只有 1 张、无 href）回退到 V3-style direct image stack：

- 新增 `_is_plain_vertical_stack(groups)` 判断普通链路。
- 新增 `_build_v3_plain_image_stack(groups, display_w, is_base64)`。
- `assemble_html()`：普通链路使用连续 `<img>`，hotspot/复杂链路保留原 table 分组结构。
- `generate_plain_html()`：复制 HTML 普通链路也使用同一 V3 结构。

## 修复后结构取证

3 张普通切片生成 HTML：

```text
div count        = 0
table count      = 1
tr count         = 1
td count         = 1
img count        = 3
has height attr  = False
```

## 风险

- 🟡 普通链路回退到 V3 HTML 结构，理论上最接近历史无缝版本。
- 🟡 仍保留 materialize 当前行为：650 会被归一到 652，PNG 宽度也会变为 652。这是第二嫌疑变量，本次未同时修改，避免一次改两个变量。
- 🟢 hotspot/带 href 复杂链路暂不改，避免扩大改动面。
